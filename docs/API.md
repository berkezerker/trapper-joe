## Command Overview

Admin commands allow you to manage traps remotely via Meshtastic mesh messages.

**Command Format:**
```
trap -<command> [options]
```

**Requirements:**
- Sender node ID must be registered in `trap_state.json`
- Commands must start exactly with "trap -" (case-insensitive for command part)
- Spaces matter! Use exact format shown

**Available Commands:**
- `trap -register [name]` — Register new trap
- `trap -status` — Query trap status
- `trap -reset` — Manually reset trap state to OK
- `trap -remove` — Unregister trap from monitoring
- `trap -statusmail` — Request immediate status report email

## Admin Commands

### trap -register [name]

**Purpose:** Register a new trap for monitoring

**Syntax:**
```
trap -register MyTrap1
```

**Parameters:**
- `name` (optional) — Display name for trap (default: sender node ID if omitted)

**Behavior:**
1. Creates new entry in `trap_state.json` if not exists
2. Sets state to "OK"
3. Initializes all metrics to empty/default
4. Sends confirmation email with full status report
5. If already registered, updates name only

**Response Email:**
- Subject: `✅ Falle registriert: MyTrap1`
- Body: Current status report

**Example Registry Entry:**
```json
"^all": {
  "name": "MyTrap1",
  "state": "OK",
  "active": true,
  "battery": null,
  "voltage": null,
  "rssi": null,
  "snr": null,
  "lastEventType": null,
  "last_processed_ts": 1712960000
}
```

**Use Cases:**
- First-time trap setup
- Adding new trap to network
- Renaming existing trap (just send register with new name)

---

### trap -status

**Purpose:** Query current trap status

**Syntax:**
```
trap -status
```

**Parameters:** None

**Behavior:**
1. Looks up trap in `trap_state.json`
2. Prints status to console (notifier terminal)
3. No email sent (use `-statusmail` for that)
4. Returns immediately

**Console Output Example:**
```
----- Status MyTrap1 -----
State: OK
Last heard: 2024-04-12 15:30:45
Battery: 85%
Voltage: 4.15V
Last Event: DETECTION (2024-04-12 15:25:00)
```

**Use Cases:**
- Quick status check during fieldwork
- Debugging connectivity issues
- Manual verification during testing

---

### trap -reset

**Purpose:** Manually reset trap from ALERT to OK state

**Syntax:**
```
trap -reset
```

**Parameters:** None

**Behavior:**
1. Sets trap state from "ALERT" to "OK"
2. Clears alert condition
3. Sends confirmation email
4. Useful when trap was checked and rebaited

**Response Email:**
- Subject: `✅ Falle manuell zurückgesetzt: MyTrap1`
- Body: Status report showing OK state

**State Change:**
```json
// Before
{ "state": "ALERT", "lastEventType": "DETECTION" }

// After
{ "state": "OK", "lastEventType": null }
```

**Use Cases:**
- Manually resetting trap after check
- Clearing false alarm (if needed)
- Re-enabling monitoring after catch

---

### trap -statusmail

**Purpose:** Trigger immediate status report email

**Syntax:**
```
trap -statusmail
```

**Parameters:** None

**Behavior:**
1. Forces immediate status report email
2. Bypasses normal schedule times
3. Useful for urgent updates
4. Can be used multiple times per day

**Response Email:**
- Subject: `📊 Manueller Statusreport`
- Body: Full status of all traps

**Distinction from Scheduled Reports:**
- Scheduled: Automatic, configurable times (e.g., 06:45, 19:00)
- Manual: On-demand, anytime via this command

**Use Cases:**
- Quick update during emergency
- Verifying system operation during testing
- Urgent status check request

---

### trap -remove

**Purpose:** Unregister and stop monitoring a trap

**Syntax:**
```
trap -remove
```

**Parameters:** None

**Behavior:**
1. Completely removes trap from `trap_state.json`
2. Stops all monitoring for this trap ID
3. Sends confirmation email with remaining traps
4. Cannot be undone (will need to re-register)

**Response Email:**
- Subject: `🗑️ Falle entfernt: MyTrap1`
- Body: Updated status report (this trap gone)

**Use Cases:**
- Decommissioning trap for season
- False node ID (remove and re-register with correct ID)
- Simplifying deployment (fewer traps to monitor)

⚠️ **Warning:** This is permanent until re-registered. No undo!

## Message Formats

### Trap Detection Message

**Source:** Trap sensor broadcasting trap trigger

**Format Options:**

**A. Sensor App Message (Recommended)**
```
DETECTION_SENSOR_APP (portnum = 4)
Message: "trap detected" (auto-generated by sensor)
```

**B. Text Message**
```
TEXT_MESSAGE_APP (portnum = 1)
Message: "trap detected"
```

**Processing:**
1. Listener receives portnum 4 or "detected" in text
2. Sets trap state to "ALERT"
3. Triggers alert email immediately
4. Continues monitoring

**JSON in messages_log.jsonl:**
```json
{
  "ts": 1712960123.456,
  "id": "^all",
  "msg": "trap detected",
  "rssi": -105,
  "snr": 8.5,
  "battery": 85,
  "voltage": 4.15
}
```

**Alert Email Triggered:**
- Subject: `🚨 ALERT: MyTrap1`
- Contains: Trap details, signal strength, immediate actions

---

### Trap Reset Message

**Source:** Manual reset button on trap or admin command

**Format:**
```
TEXT_MESSAGE_APP (portnum = 1)
Message: "trap reset"
```

**Processing:**
1. Listener receives "trap reset" message
2. Sets trap state to "OK"
3. Clears "lastEventType"
4. Sends reset confirmation email

**JSON in messages_log.jsonl:**
```json
{
  "ts": 1712960150.123,
  "id": "^all",
  "msg": "trap reset",
  "rssi": -102,
  "snr": 9.0,
  "battery": 84,
  "voltage": 4.14
}
```

**Reset Email Triggered:**
- Subject: `🔄 Reset: MyTrap1`
- Contains: Reset timestamp, new OK status

---

### Telemetry Message

**Source:** Device sending periodic metrics

**Format:**
```
TELEMETRY_APP (portnum = 67)
Content: device metrics (battery, voltage, temperature, etc.)
```

**Extracted Data:**
```json
{
  "ts": 1712960160.789,
  "id": "^all",
  "type": "telemetry",
  "battery": 85,
  "voltage": 4150,
  "temperature": 22.3,
  "channelUtilization": 12.5,
  "airUtilTx": 2.3,
  "uptimeSeconds": 86400
}
```

**Processing:**
- Not converted to "message"
- Logged to separate telemetry_log.jsonl
- Used to update device metrics in trap state
- No email triggered (informational only)

## Alert Types

TrapperJoe sends different email types based on events:

### 1. 🚨 TRAP ALERT Email

**Triggered By:** Trap detection

**Subject:** `🚨 ALERT: [TrapName]`

**When Sent:** Immediately when "trap detected" received

**Contents:**
- ⭐ Visual highlighting of alert
- Trap name and ID
- Time of detection
- Signal strength (RSSI, SNR) with quality rating
- Battery level and voltage
- Count of other active alarms
- Suggested next steps

**HTML Styling:**
- Red/pink gradient header
- Alert box with warning icon
- Table with trap details
- Recommendation for user action

**Example:**
```
Subject: 🚨 ALERT: Trap1

[RED HEADER] ALARM AUSGELÖST
Eine Falle hat eine Detektion gemeldet

[ALERT BOX] Alert: Falle Trap1 hat am 12.04.2024 um 15:30:45 
eine Detektion gemeldet.

[DETAILS TABLE]
Falle ID:    ^all
Status:      ALERT
Zeitpunkt:   2024-04-12 15:30:45
Batterie:    85%
Spannung:    4.15V
RSSI:        -105 dBm (Schwach)
SNR:         8.5 dB

[ACTION ITEMS]
1. Falle überprüfen und Fang kontrollieren
2. Falle zurücksetzen mit: trap reset
3. Status abrufen mit: trap -statusmail
```

---

### 2. ⚠️ MISSING/TIMEOUT Email

**Triggered By:** Device offline longer than `alive_timeout_hours`

**Subject:** `⚠️ Timeout: [TrapName], ...`

**When Sent:** Once per offline period (no duplicates)

**Contents:**
- Orange/yellow gradient header
- List of offline traps
- Last heard timestamp
- Troubleshooting suggestions:
  - Low battery
  - Out of range
  - Hardware failure
  - Weather impact
- Recommendation to check on-site

**Example:**
```
Subject: ⚠️ Timeout: Trap1, Trap3

[ORANGE HEADER] VERBINDUNGS-TIMEOUT
2 Falle(n) seit 1h nicht mehr gehört

[WARNING BOX] Die folgenden Fallen haben das Timeout von 1 Stunden 
überschritten und senden keine Nachrichten mehr.

[OFFLINE TRAPS]
• Trap1 (^all) - Zuletzt gehört: 12.04.2024 14:30:00
• Trap3 (^bcd) - Zuletzt gehört: 12.04.2024 14:25:00

[POSSIBLE CAUSES]
• Batterie leer oder kritisch niedrig
• Falle außerhalb der Funkreichweite
• Hardware-Defekt oder Beschädigung
• Extreme Wetterbedingungen

[RECOMMENDED ACTION]
Falle(n) vor Ort überprüfen
```

---

### 3. ✅ RESET Email

**Triggered By:** `trap reset` command or manual reset

**Subject:** `✅ Falle manuell zurückgesetzt: [TrapName]`

**When Sent:** Immediately after reset

**Contents:**
- Green gradient header
- Trap name and ID
- Reset timestamp
- New state: OK
- Confirmation message

**Example:**
```
Subject: ✅ Falle manuell zurückgesetzt: Trap1

[GREEN HEADER] RESET BESTÄTIGUNG
Falle wurde erfolgreich zurückgesetzt

[DETAILS]
Falle ID:     ^all
Falle Name:   Trap1
Status:       OK (zurückgesetzt)
Reset Zeit:   2024-04-12 15:35:00

[CONFIRMATION]
✅ Die Falle ist nun wieder betriebsbereit und überwacht aktiv.
```

---

### 4. 📊 STATUS REPORT Email

**Triggered By:** Scheduled times (e.g., 06:45, 19:00) or `trap -statusmail`

**Subject:** `📊 Täglicher Statusreport`

**When Sent:** Scheduled times (once per day), or on demand

**Contents:**
- Summary cards (OK count, ALERT count, MISSING count)
- Table of all traps with:
  - Name and ID
  - Current state (OK/ALERT/MISSING)
  - Last heard
  - Battery level
  - Signal strength
  - Last event type
- Recent events timeline
- System health summary

**Example:**
```
Subject: 📊 Täglicher Statusreport

[SUMMARY CARDS]
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│   OK        │  │   ALERT     │  │  MISSING    │
│   18        │  │   2         │  │  1          │
└─────────────┘  └─────────────┘  └─────────────┘

[TRAP STATUS TABLE]
Name      │ State   │ Last Heard       │ Battery │ Signal
Trap1     │ OK      │ 12.04 14:30      │ 85%     │ -105dBm
Trap2     │ ALERT   │ 12.04 15:30      │ 92%     │ -98dBm
Trap3     │ MISSING │ 12.04 10:15      │ 45%     │ N/A
...

[RECENT EVENTS]
12.04 15:30 - Trap2: Detection
12.04 15:25 - Trap1: Status update
12.04 14:45 - Trap3: Timeout (offline 1h+)
```

## Status Report Format

The status report email body is an HTML table with the following columns:

```
┌─────────────────────────────────────────────────────────────────────┐
│ Trap Name │ State   │ Last Heard      │ Battery │ RSSI    │ Event  │
├─────────────────────────────────────────────────────────────────────┤
│ Trap1     │ ✅ OK   │ 12.04 14:30:45  │ 85%     │ -105dBm │ Reset  │
│ Trap2     │ 🚨 ALERT│ 12.04 15:30:20  │ 92%     │ -98dBm  │ Detect │
│ Trap3     │ ⚠️ MISSING│ 12.04 10:15:00 │ 45%     │ N/A     │ Timeout│
└─────────────────────────────────────────────────────────────────────┘
```

**State Colors:**
- 🟢 ✅ OK (Green) — Normal operation
- 🔴 🚨 ALERT (Red) — Recent detection
- 🟡 ⚠️ MISSING (Orange) — Device offline

**Battery Levels:**
- 🔋 75-100% (Green) — Good
- 🟡 50-74% (Yellow) — Fair
- 🔴 0-49% (Red) — Low, may need replacement soon

**RSSI Quality:**
- 📶📶📶 -40 to -80 dBm (Green) — Excellent
- 📶📶 -80 to -100 dBm (Yellow) — Good
- 📶 -100 to -120 dBm (Red) — Fair/Poor
- ❌ <-120 dBm (Red) — Unusable
